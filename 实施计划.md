# 研究生科研工作台实施计划

## 1. 项目定位

本系统建议定位为“研究生科研工作台”，面向研究生、课题组成员和实验室管理员，统一承接以下工作流：

- 找资料：学术检索、网页检索、历史回看
- 沉淀资料：知识库归档、标签化、笔记化、RAG 问答
- 生成结果：总结、提示词优化、Bio 医学助手、研究报告
- 推进任务：番茄钟、计划清单、日历、打卡
- 个人协作：个人主页、微信/飞书绑定、项目空间共享

结合现有 `pom.xml` 与 `application.yml`，当前后端能力已经具备：

- `Spring Boot 3.2 + Java 17 + JPA + Security + JWT`
- `MySQL 8 + Redis`
- 文件上传与本地目录存储
- OpenAI 兼容接口、阿里云 DashScope、MetaSo 搜索接入配置

这意味着系统可以优先做成一个“研究工作台型 Web 产品”，而不是单点工具页。

## 2. 现状判断与约束

### 2.1 已具备条件

- 已有 Spring Boot 项目骨架和主要依赖
- 已在配置中预留 AI、MetaSo、Redis、JWT
- 已有明确的一级功能清单
- 已提供前端视觉参考图，适合走“轻科技 + 学术工作台”风格

### 2.2 当前需要补齐的问题

- `pom.xml` 中项目名仍是 `cancer-system`，与当前“科研工作台”定位冲突，建议统一重命名
- `application.yml` 目前允许上传的文件类型偏向 `csv/tsv/txt/video`，不够支撑论文场景，建议扩展 `pdf/doc/docx/md/pptx`
- 当前没有看到前端工程目录，建议单独建立 `Vue 3 + Vite` 前端项目
- 当前配置文件中存在数据库密码与 API Key 明文，正式开发前必须改为环境变量
- 当前未体现异步任务中心，后续所有“搜索/总结/入库/生成报告”都需要任务化

## 4. 产品范围建议

## 4.1 MVP 范围

第一阶段不要把所有功能同时做满，建议先收敛到 6 个核心模块：

1. 登录与个人中心
2. 学术调研
3. 知识库
4. 网页搜索
5. AI 服务
6. 研究计划

## 4.2 MVP 必做功能

### A. 学术调研

- 研究问题输入
- 基于 MetaSo / 外部学术源的结果聚合
- 结果卡片：标题、摘要、来源、发表时间、作者、标签
- 一键总结
- 一键加入知识库
- 搜索历史
- 收藏与项目归档

### B. 知识库

- 知识库创建、重命名、删除
- 文件夹管理
- 文件上传、批量删除、标签管理
- 文档解析与分段
- 知识库问答
- 知识库摘要
- 文档引用回溯

### C. 网页搜索

- 通用网页搜索
- 抖音 / 小红书 / 网站内容聚合卡片
- 结果转 Markdown
- 保存到项目或知识库
- 搜索历史

### D. AI 服务

- 提示词优化器
- Bio 医学助手
- 常用 AI 工具导航
- 会话记录
- 模板化输出

### E. 研究计划

- 工作清单
- 番茄钟 / 计时器
- 日历事件
- 每日打卡
- 进度统计

### F. 个人中心

- 个人资料编辑
- 头像、简介、研究方向
- 微信 / 飞书绑定
- 我的收藏、我的搜索、我的产出

## 4.3 第二阶段增强功能

- 项目空间 Workspace
- 文献关系图谱
- 研究订阅提醒
- 报告中心
- 学习卡片 / Quiz
- PDF 高亮批注
- Zotero/BibTeX/RIS 导入导出
- 团队协作与共享

## 5. 推荐系统架构

## 5.1 前端

建议：

- Vue 3
- Vite
- Vue Router
- Pinia
- Element Plus
- Axios
- ECharts
- Sass

前端模块划分：

- `modules/auth` 登录与权限
- `modules/dashboard` 首页工作台
- `modules/research` 学术调研
- `modules/kb` 知识库
- `modules/web-search` 网页检索
- `modules/ai` AI 服务
- `modules/plan` 研究计划
- `modules/profile` 个人中心
- `components/common` 通用卡片、筛选器、时间线、Markdown 预览

## 5.2 后端

按业务域拆分为：

- `auth` 用户认证、JWT、第三方绑定
- `workspace` 项目空间
- `research` 学术检索、保存、引用、历史
- `knowledge` 知识库、文档、分段、RAG
- `search` 网页搜索、MetaSo 聚合
- `assistant` AI 会话、提示词模板、工具目录
- `plan` 任务、日历、番茄钟、打卡
- `artifact` 报告、脑图、学习卡片、Markdown 导出
- `file` 文件上传、解析、预览

## 5.3 核心后端能力建议

### 检索层

- MetaSo 作为统一网页检索入口
- 学术搜索结果建议标准化为统一结构，屏蔽不同来源差异
- 搜索结果全部落库，支持二次总结和历史回放

### 知识库层

- 文档元数据入库
- 文本切块入库
- 初期可先用 MySQL 全文检索 + 标签筛选
- 二期再增加向量检索

### AI 编排层

- 对 OpenAI 兼容接口和 DashScope 做统一适配层
- 提供 `chat`、`summary`、`rewrite`、`extract`、`report` 五类标准能力
- 所有生成结果必须记录 `prompt`、`model`、`source_refs`

### 任务层

- 所有耗时任务做成异步任务
- 任务状态：`PENDING/RUNNING/SUCCESS/FAILED/CANCELED`
- 对搜索、解析、总结、生成报告统一复用任务中心

## 6. 页面与业务流程建议

### 6.1 主流程

1. 用户登录进入首页工作台
2. 选择“学术调研”发起研究问题
3. 查看搜索结果并筛选
4. 保存高价值结果到项目空间或知识库
5. 基于保存内容发起 AI 总结/脑图/报告
6. 将下一步研究动作加入计划中心

### 6.2 关键闭环

闭环一：搜索 -> 保存 -> 知识库问答 -> 生成报告  
闭环二：网页搜索 -> Markdown 摘要 -> 保存到项目 -> 后续追踪  
闭环三：研究计划 -> 番茄钟 -> 打卡 -> 数据统计

## 9. 风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 配置泄漏 | `application.yml` 明文密码和 Key | 改为环境变量、`application-local.yml`、部署密钥托管 |
| 文件类型不足 | 当前不支持 PDF/Word | 扩展上传白名单与解析器 |
| AI 结果不可追溯 | 生成内容缺乏来源 | 所有摘要保留 `source_refs` 与引用片段 |
| 耗时接口阻塞 | 搜索、解析、总结时间长 | 统一异步任务中心 |
| 产品过宽 | 功能多但都不深 | 先做 MVP 闭环，再做增强能力 |
| 命名混乱 | 现工程名与产品不一致 | 统一包名、数据库名、前端品牌名 |

## 10. 结论

这套产品完全可以实施，而且不需要推翻当前技术栈。最合理的方式不是一次做满，而是围绕“研究问题驱动”的主流程推进：

- 第一优先级：搜索、保存、知识库、总结
- 第二优先级：项目空间、计划系统、个人沉淀
- 第三优先级：图谱、订阅、协作、导出生态

如果按以上分期推进，6 到 8 周可以做出可用版本，10 到 12 周可以进入较完整的研究工作台形态。
